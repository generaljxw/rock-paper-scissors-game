## 1. 目的与范围

### 1.1 目的
本文档定义了猜拳小游戏项目文档的版本控制标准，确保项目文档的变更可追溯、版本状态清晰，支撑敏捷开发流程的有效运作。

### 1.2 适用范围
- 项目需求文档（PROJECT_BRIEF.md）
- 特性需求清单（FEATURE_SPEC.md）
- 架构设计文档（docs/ad/*.md）
- 详细设计文档（docs/dd/*.md）
- 开发指南（DEVELOPMENT_GUIDE.md）
- 用户手册（docs/manuals/*.md）

## 2. 版本号规范

### 2.1 版本号格式
```
主版本号.次版本号.修订号
```

| 组成部分 | 说明 | 示例 |
|----------|------|------|
| 主版本号 | 重大架构变更或版本里程碑 | v1.0.0 → v2.0.0 |
| 次版本号 | 新增功能模块或较大变更 | v1.0.0 → v1.1.0 |
| 修订号 | 小的修正或文档优化 | v1.0.0 → v1.0.1 |

### 2.2 版本状态定义

| 状态 | 标识 | 说明 |
|------|------|------|
| 草稿 | Draft | 文档正在编写中 |
| 待评审 | Pending Review | 文档完成，等待评审 |
| 评审中 | Under Review | 正在进行评审 |
| 已确认 | Confirmed | 评审通过，待基线化 |
| 基线版本 | Baseline | 正式交付的基准版本 |
| 已归档 | Archived | 被新基线替代的旧版本 |

## 4. 文档变更管理流程

### 4.1 变更申请流程
```
┌─────────────────┐
│ 识别需要变更    │
└────────┬────────┘
         │
         ▼

│ 填写变更申请单  │──── 不批准 ────> 结束
└────────┬────────┘
         │ 批准
         ▼
┌─────────────────┐
│ 进行文档修改    │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 提交评审        │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 评审通过?       │
└────────┬────────┘
    是   │   否
    │    │    │
    ▼    │    ▼
```

## 5. 基线版本管理

### 5.1 基线版本创建条件
- 文档通过完整评审流程
- 获得所有必要签字
- 文档内容完整、准确

### 5.2 基线版本标识
基线版本的文档必须：
1. 文档状态标记为"基线版本"
2. 包含基线版本号
3. 包含基线创建日期
4. 包含基线保存路径

### 5.3 基线版本保存
- 基线版本保存到 `docs/[文档类型]/baseline/` 目录
- 文件命名格式：`[文档名]_v[版本号].md`

## 6. 评审流程

### 6.1 评审角色
| 角色 | 职责 |
|------|------|
| 作者 | 编写文档，提交评审 |
| 项目负责人 | 最终审批 |
| 相关Agent | 专业评审 |

### 6.2 评审检查清单
- [ ] 文档格式符合规范
- [ ] 内容完整、准确
- [ ] 与其他文档一致
- [ ] 无错别字或语法错误
- [ ] 版本控制信息完整

### 6.3 评审签字要求
| 文档类型 | 必要签字 |
|----------|----------|
| 需求文档 | 项目负责人、Planner |
| 架构设计 | 项目负责人、Planner、Coder |
| 详细设计 | 项目负责人、Planner、Coder、Tester |
| 测试文档 | 项目负责人、Tester |
